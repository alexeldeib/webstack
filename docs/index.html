webstack

A project to prototype web stacks github.com/croaky/webstack

API checks from N. California, Montreal, Ireland, Stockholm, and Sydney:

overview

When providing a JSON API, we developers choose a hosting provider, runtime, and SQL database. This project is my attempt to prototype options for each.

Right now, I'm focused on these hosting providers:

  • Fly.io
  • Railway
  • Render

and these runtimes (frameworks):

  • Go (stdlib)
  • Node (Express)
  • Ruby (Sinatra)

with these SQL databases:

  • Postgres

Each stack has a healthcheck-style API endpoint that executes a SELECT 1 to a SQL database and responds with JSON { "status": "ok" }.

serverless

I tested two serverless options: Deno Deploy and Vercel's serverless Go functions. Both used Supabase as the backend database.

I'm still intrigued by the future of serverless. But, serverful options are a little more "standard"; the same code is likely to run on Fly, Railway, Render, or Heroku without many tweaks. The latency is also higher for the serverless options:

heroku + ruby + postgres

From 2009-2017 and 2020-2022, my go-to stack was Heroku + Ruby + Postgres.

heroku

My Heroku setup was generally:

  • Version code in a GitHub repository
  • Open GitHub pull requests w/ Continuous Integration (CI)
  • Merge into main branch to auto-deploy to Heroku staging environment
  • Promote staging manually to production environment in a pipeline
  • Configure Postgres database with a follower

Heroku's GitHub integration broke April 2022 in incident 2413 and has not been restored. Heroku also lacks overdue features that Render now supports:

  • HTTP/2 support
  • IP access control on Postgres databases
  • DDoS protection
  • Health checks
