example

Each stack serves a healthcheck-style HTTP API endpoint that executes a SELECT 1 to a SQL database and responds with JSON { "status": "ok" }. Each stack is built with a lightweight router, database connection pool, and SQL database driver (no ORM). For example, in Ruby: 

      
require "connection_pool"
require "pg"
require "sinatra"

class DB
  def initialize
    @pool = ConnectionPool.new(size: 5, timeout: 5) {
      PG.connect(ENV.fetch("DATABASE_URL"))
    }
  end

  def exec(sql)
    @pool.with do |conn|
      conn.exec(sql)
    end
  end
end

db = DB.new

get "/" do
  db.exec "SELECT 1"
  content_type :json
  {status: "ok"}.to_json
end

heroku

From 2009-2022, my stack most often included Heroku + Postgres:

  • Version code in a GitHub repo
  • Open GitHub pull requests w/ CI
  • Merge into main branch to auto-deploy to staging environment
  • Promote staging manually to production environment in a pipeline
  • Configure production database with a follower

Heroku's GitHub integration broke April 2022 in incident 2413 and has not been restored. Heroku also lacks overdue features that e.g. Render now supports:

  • HTTP/2 support
  • IP access control on Postgres databases
  • DDoS protection
  • Health checks

serverless

For Deno Deploy and Vercel serverless Go function, I used Supabase for the database.

Serverless feels like it could be the future; it would be nice to manage fewer things. I also like the web standards focus of Deno.

My higher priority right now, thought, is "lifting and shifting" traditional Ruby, Node, or Go backends from Heroku to Encore, Fly, Railway, or Render.
